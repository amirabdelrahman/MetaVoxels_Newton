Control design note, rev. 2

Phase-coded oscillators for shape × control
co-design on MetaVoxels.

Each actuator is assigned one slot from a three-entry table of phase offsets. At simulation time every actuator drives a sinusoidal target with its assigned phase. One integer per actuator, fixed for the whole rollout, shared across locomotion, climbing, and grasping. The flat RL action space grows by a single multiplicative factor K when this option is turned on, not by a per-timestep control loop.

01What changed, and why

The previous draft of this file proposed a library of named whole-body poses (GAIT_A, GAIT_B, BRACE, GRASP). That is a reasonable choice for a fixed-morphology robot, but it is the wrong shape for co-design. When the morphology is itself being searched over, a named pose like GAIT_A only makes sense if the body it was written for still exists. Evolving the shape invalidates the library.

The right abstraction is one that attaches to each actuator individually, emerges coordination from the relative timings, and makes no assumption about body topology. Phase-coded oscillators do exactly that. They are also the control primitive used in most of the voxel robot and soft robot literature for this reason.

02The model

targeti(t)  =  A · sin( 2π ( fi · t  +  ϕi ) ) A is a global amplitude. Each actuator i chooses its own phase ϕi and frequency multiplier fi from a small discrete table.

The default table has K = 3 phases and L = 1 frequency, so each actuator picks one of three slots. This is the minimum that produces non-trivial group behavior. Raising K to four or L to two is a one-line change when the task needs it.

03How the action space grows

Shape only

|A|  =  1  +  T · n³
T voxel types, n³ grid cells, plus a STOP action.

Shape × control

|A|  =  1  +  Tpassive · n³  +  Tactuator · n³ · KL
Only actuator placements are multiplied by KL. Passive types are untouched.

For n = 7, Tpassive = 2, Tactuator = 2, K = 3, L = 1:

1  +  2 · 343  +  2 · 343 · 3  =  1  +  686  +  2058  =  2745 actions Compared to 1373 without control choice. A 2x jump, not exponential. MaskablePPO handles this range fine.

04What each phase table means

Three sine waves shifted by one third of a period each. Actuators in the same slot move together. Actuators in different slots are offset by 120 or 240 degrees. That is already enough to produce rhythmic group patterns.

05Why one scheme covers three tasks

Locomotion
Assign alternating slots to diagonal actuator pairs. Pairs in slot 0 lift while pairs in slot 1 push. Reverse the pattern after half a period and the body translates.
Climbing
Put a load-bearing subset in slot 0 and reach with the rest in slot 2. The phase offset guarantees at least part of the body is always anchored.
Grasping
Assign every actuator to the same slot. They contract in unison, closing around an object. Add a tiny phase delta on the fingertips for a whip-like catch.

The RL agent is never told which task it is solving. It just searches over phase assignments that happen to score well under the rollout reward. Different assignments emerge for different bodies.

06Alternatives considered

Approach Action-space growth Works across bodies Interpretable Verdict
Named whole-body poses constant no yes Breaks whenever the morphology changes. Bad fit for co-design.
Continuous torques per actuator per step huge yes no Hard to mask, hard to read, expensive to explore.
Central pattern generators (learnable) medium yes partial Good but introduces a second optimization layer.
DMPs / learned motor primitives medium partial yes Powerful but requires warm examples per body.
Phase-coded oscillator slots (K=3, L=1) linear in KL yes yes Minimal, general, trivially maskable, survives shape changes.

07Why this is a good fit for MaskablePPO

One integer per actuator

The slot is a discrete choice made at placement time. It lives next to the shape choice in the flat action index.

Masking is a slice

Disabling actuator type t's slot m is a contiguous range of n³ action ids. The layout helper computes the offsets.

No per-timestep control head

The policy decides the slot once per actuator and never again. Control does not enter the rollout loop as an action.

Replicates at zero cost

Copying the bank and DOF index list to N worlds is a buffer copy. The MuJoCo-Warp solver does the rest.

Clean reward shaping

Forward speed, upright cosine, height delta and more come straight out of the RolloutStats object.

Exploration stays honest

Only KN distinct phase patterns exist for N actuators. Random exploration actually samples the space.

09What is not in this file

The RL side of the co-design loop. MaskablePPO, the env, the reward shaping, and the eval callbacks live in a separate module. The control library only provides the physics-side primitives: the oscillator bank, the joint controller, and the action-index helpers. That separation keeps the sim and the learning loop independently testable.

A follow-up note will walk through a modernized MaskablePPO setup (recent tricks: action-masking in the loss, on-policy critic targets with variance reduction, DAPO-style length bias fixes, token-level advantage normalization, plus entropy scheduling suited to discrete action spaces with large masks). That note will reference the layout object above so masks and actions stay in sync.
